## 1. 产品定位

### 产品名称

家庭医疗记忆系统（Family Health Memory）

### 产品愿景

建立一个长期积累的家庭医疗记忆库。让家庭成员过去的就诊记录、检查报告、化验单、处方信息、药品档案，能够在几年后依然快速检索和查看。

**产品哲学**：医疗领域里**记忆比推理更有价值**。家庭真正需要的往往不是 AI 给出的诊断，而是 3 年前那份检查报告、那次药物过敏记录、那张出院小结。本系统的核心价值是长期可靠的医疗记忆，AI 只是整理记忆的工具。

**系统双重身份**（v3.1 明确）：既是**家庭医疗记录器**（Capture：先记录后整理），也是**家庭医疗档案库**（Archive：长期检索）。v3 偏后者，v3.1 补齐前者。

---

## 2. 产品目标

构建统一的家庭医疗档案库，实现：快速记录（Capture）、医疗资料统一归档、医疗事件时间线管理、药箱清单管理、AI 辅助摘要 + OCR 全文、关键词搜索（MVP）/ AI 语义搜索（第二阶段）。

---

## 3. 使用场景与约束

### 3.1 使用场景

- **当前定位**：单家庭自用，1～2 名管理员维护，不对外发布不商业化
- 数据规模：家庭成员 2～6 人，累计事件数百到数千条，原件数千份

### 3.2 现阶段取舍：AI 接入优先于隐私合规

**决策**：当前阶段优先保证 AI 能力可用，不投入精力做隐私合规框架。

**含义**：

- 上传的医疗资料（图片/PDF）直接发送至第三方 AI 服务完成 OCR、多模态识别、摘要
- 图片上的姓名、身份证号、就诊卡号等 PII 会随原图上云（自用场景已接受）
- 通过 prompt 约束 AI 不把 PII 写入摘要和标签，降低结构化泄露面
- 不做数据流向弹窗、不做本地 OCR 兜底、不做端到端加密

**重新评估时机**：若未来分享给亲友或公开发布；若成员资料涉及非直系亲属明示不同意上云。

### 3.3 数据安全底线（与隐私无关，必须做）

数据本地存储为主，支持完整导出可迁移，清缓存/换设备/手机丢失场景数据可恢复。详见 7.9。

## 4. 产品边界

不做：在线问诊、疾病诊断、自动开药、药品销售、医院系统对接、全国医院格式适配、多用户权限/分享。

---

## 5. 核心设计原则

- **原件优先**：保留原始文件，AI 失败不影响查看
- **医疗事件驱动**：核心对象是 Medical Event（接近 EMR Encounter 模型）
- **AI 只负责整理**：OCR/摘要/标签/搜索辅助；不做诊断/用药/决策
- **AI 内容可信度标注**：标注"以原件为准"，关键数值保留原文
- **数据架构面向长期**：schema 宁可现在多花时间做对（独立表/多对多/抽象引用），不埋日后重构的雷
- **Capture First**（v3.1 新增）：记录的时机稍纵即逝（孩子夜里发烧），系统必须支持"先记录后整理"，不能假设用户每次都完整录入

---

## 6. 用户角色

家庭管理员（1～2 人，无权限模型）+ 家庭成员（档案归属对象）。

### 7.1 Quick Capture 快速记录箱（v3.1 新增，MVP 最高优先级）

**定位**：Capture First。整个系统最前置的入口。真实场景是孩子夜里 39 度，家长先拍照、以后再整理——而不是创建事件填一堆字段。

**为什么优先级最高**：v3 偏"档案管理系统"，假设用户每次都完整录入。但真实记录行为是"先记下来，有空再整理"。缺快速入口，用户会回到微信/相册（那里拍照最快），系统失去捕获时机——后续所有归档、检索、AI 都成了无源之水。

**功能**：

- 首页顶部"+ 快速记录"按钮（比"+ 新建事件"更显眼、更快）
- 点击直接进入捕获模式，三选一：
  - **拍照**：调用相机，可连拍多张（病历/化验单/药盒）
  - **语音**：录一段口述（"大宝昨晚 39 度，吃了美林退到 37.5"），存音频
  - **文字**：一句话速记
- 保存到 Inbox（收件箱），状态 = 未归档
- **不强制填**任何字段（成员/医院/科室都可空，归档时再补）
- 单次捕获目标 ≤ 15 秒

**Inbox 整理流程**：

- Inbox 项可后续"归档"为正式医疗事件：补全成员/日期/医院等字段
- 归档时：inbox_item 的媒体转为 attachment，关联到新建或已有 event
- 归档后 inbox_item 标记为已归档（保留引用，不删除）

**Dashboard 提醒**：首页顶部显示"📥 N 条待整理"，未归档 Inbox 数 > 0 时高亮。

### 7.4 医疗事件管理（v3.1 重构：AI 辅助健康问题关联）

#### 字段

日期、成员、医院、科室、标题、类型、备注、健康问题（关联）。

#### 事件类型

门诊、急诊、体检、复查、疫苗、住院、其他。

#### 健康问题关联（v3.1 重构交互）

底层保留 `health_problems` 表 + `event_problem_rel` 多对多（长期正确）。但 **MVP 不做手工 health_problems CRUD**——用户不会认真维护状态，会变空表，且手工关联会让录入超 2 分钟。

**MVP 交互**（AI 辅助，一键确认）：

1. 创建事件时，用户**可选填**一个健康问题自由文本（字符串，快速，可空）
2. 事件保存后，AI 基于事件标题/摘要/OCR 全文，推荐关联：
   ```
   发现可能关联的健康问题：
   ◉ 反复呼吸道感染（已有，3 次关联）
   ○ 新建：哮喘随访
   
   [ 确认关联 ]  [ 跳过 ]  [ 稍后在 Inbox 整理 ]
   ```
3. 用户一键确认 / 跳过 / 稍后整理
4. health_problems 表通过这种方式**渐进积累**，无需手工维护

**为什么这样**：表结构保留架构正确（v3 的判断不变），但交互从"手工 CRUD"改为"AI 推荐 + 一键确认"，既不增加录入负担，又让 health_problems 表逐步填充。状态字段（进行中/已结束）留待第二阶段在健康问题详情页维护。

### 7.5 医疗资料归档（v3.1 加 processing_status 状态机）

#### 支持上传

JPG / PNG / PDF（从 Quick Capture 归档，或事件页直接上传）

#### 资料类型

门诊病历、检查报告、化验单、出院小结、收费单据、医嘱单。

#### 存储内容

| 字段 | 说明 |
|------|------|
| 文件名称 | 原始名称 |
| 上传时间 | 自动记录 |
| 所属事件 | Event ID |
| storage_key | 存储抽象键（Storage Adapter 管理） |
| 标签 | 最新标签快照（JSON 数组） |
| **processing_status** | v3.1 新增：处理状态机 |
| AI 产出 | 关联 ai_contents 表 |

#### processing_status 状态机（v3.1 新增）

AI 处理是异步的，必须让用户看到进度和失败原因：

| 状态 | 含义 |
|------|------|
| `UPLOADED` | 已上传，待处理 |
| `OCR_PROCESSING` | OCR 进行中 |
| `OCR_DONE` | OCR 完成，摘要生成中 |
| `SUMMARY_DONE` | 全部完成 |
| `FAILED` | 处理失败（记录原因，可重试） |

**UI**：附件列表显示状态徽章；FAILED 时显示原因 + 重试按钮。避免 AI 失败后用户不知道发生什么。

### 7.6 AI 摘要与 OCR 全文（多版本）

上传资料后自动执行。一轮调用产出 summary + ocr_fulltext + tags，存入 ai_contents 表。

#### AI 产出类型（ai_contents 表）

| content_type | 内容 | 用途 |
|------|------|------|
| `summary` | 结构化摘要 | 事件页快速浏览 |
| `ocr_fulltext` | 原始 OCR 全文 | 关键词搜索（7.8）、未来 RAG 的基础 |

**为什么存 OCR 全文**：未来搜索/RAG 全靠它；现在不存，两年后 5000 份报告要重跑 OCR。同一轮 API 调用零额外成本。

#### 强制要求

- 顶部标注"AI 生成，以原件为准"
- 数值保留原文（不转换不改写）
- 摘要和标签不含 PII（prompt 约束）
- 提供"查看原件 / 查看 OCR 全文 / 重新生成"入口（重新生成保留旧版本可对比）

### 7.8 关键词搜索（v3.1 新增，MVP）

**定位**：基于 ocr_fulltext 的 SQL LIKE 搜索，MVP 就能用。**不需要 RAG/embedding**。

**功能**：

- 顶部搜索框：输入关键词
- 在所有 attachments 的 ocr_fulltext 上做 LIKE 匹配
- 结果：匹配的附件列表（高亮关键词 + 所属事件 + 成员），点击跳转原件
- 支持按成员/时间范围过滤

**示例查询**：

```
阿莫西林    → 找到所有提到阿莫西林的报告/处方
肺炎        → 找到所有肺炎相关就诊
白细胞偏高   → 找到所有白细胞偏高的化验单
```

**为什么 MVP 就做**：v3 已存 ocr_fulltext，LIKE 搜索几乎零成本，但实用性极高。用户搜"阿莫西林/肺炎/白细胞"已经很有价值。把搜索整体推第二阶段是错的——只有**语义搜索**（"大宝去年吃过哪些抗生素"）才需要 RAG，那才是第二阶段。

### 7.9 数据持久性（含分卷预留）

#### 关键原则

备份必须包含原件（PDF/JPG）。只导 SQLite 等于只恢复元数据，原件丢了永久丢。

#### 功能

- **一键导出**：ZIP 打包（所有原件 + SQLite + JSON 元数据）；文件名 `medmemory-backup-YYYYMMDD-HHmm-{N}成员-{M}事件.zip`
- **一键导入**：校验 ZIP 结构和版本；合并/覆盖两种模式；**导入前自动快照**（先备份当前库，误操作可逆）
- **导出后完整性校验**：ZIP 内原件数 = attachments 记录数，不一致告警
- **未备份提醒**：30 天未导出 或 新增/修改 ≥20 事件未导出 时提示（非弹窗）
- **手动触发**：设置页"立即备份"

#### 未来扩展（不进 MVP）

- **分卷导出**：几年后数据可能 5GB+，浏览器一次性打包大 ZIP 可能崩溃；预留按卷（如每 500MB）分批导出
- 自动定时备份到网盘（依赖 Storage Adapter）
- 增量备份

## 9. 技术方案

### 9.1 前端

Vue3 + TypeScript + PWA

### 9.2 本地存储与 Storage Adapter

**SQLite 直接定 sqlite-wasm + OPFS**（SQL.js 判死刑——整体序列化无解）。支持增量持久化，长期性能稳定。阶段 0 验证目标浏览器兼容性。

**Storage Adapter**（抽象存储层）：

```typescript
interface StorageAdapter {
  saveFile(key: string, blob: Blob): Promise<void>
  getFile(key: string): Promise<Blob | null>
  deleteFile(key: string): Promise<void>
  listFiles(prefix?: string): Promise<string[]>
}
```

当前实现：IndexedDB（原件）、OPFS（SQLite）。未来实现：OneDrive / iCloud / WebDAV，数据模型零改动。

### 9.3 AI 能力

单一多模态大模型 API（GPT-4o / Claude Sonnet 4.5 / Gemini 2.0 Flash 候选），一轮调用产出 summary + ocr_fulltext + tags。

**prompt 约束**：保留原文数值 / 不诊断 / 输出标签 / PII 不入摘要。

**额外职责**（v3.1）：基于事件内容推荐 health_problems 关联（见 7.4）。

**第二阶段**：基于 ocr_fulltext 的 RAG 语义搜索、拍照识别药盒。
