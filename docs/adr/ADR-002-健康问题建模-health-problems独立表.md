# ADR-002: 健康问题建模为独立表 + 多对多关联

- **状态**: Accepted
- **日期**: 2026-06-21
- **决策者**: 架构评审

## 背景

v1/v2 用 `medical_events.problem_tag TEXT` 存健康问题标签（如"哮喘""反复呼吸道感染"）。

这个设计的致命问题：几年后一定会出现同义词泛滥——"发烧 / 发热 / 高热"、"上呼吸道感染 / 呼吸道感染 / 反复呼吸道感染"。字符串无法归并，**无法聚合病程**，时间线的"按健康问题视图"形同虚设。

## 决策

独立 `health_problems` 表 + `event_problem_rel` 多对多关联表，替代 `problem_tag TEXT`。

```
health_problems (id, member_id, name, status, started_date, ended_date, note)
event_problem_rel (event_id, problem_id)  -- 多对多
```

## 理由

- **可归并**：同一个健康问题（如"哮喘"）是表里的一行，无论事件标题怎么写，都能可靠聚合所有相关事件，形成真正的病程
- **接近 EMR Problem List 模型**：医疗系统的成熟实践，status（进行中/长期/已结束）可追踪疾病演变
- **多对多**：一个事件可关联多个健康问题（发烧事件同时属于"反复呼吸道感染"和"哮喘随访"），一个健康问题关联多个事件（病程）

## MVP 交互约束

MVP **不做手工 health_problems CRUD**——用户不会认真维护状态，表会变空，且手工关联会让录入超 2 分钟（违背核心指标）。

改为：事件保存后 AI 基于内容推荐关联，用户一键确认。表结构保留（长期正确），交互从"手工 CRUD"转为"AI 辅助 + 一键确认"，health_problems 表渐进积累。状态字段（进行中/已结束）留待第二阶段在健康问题详情页维护。

## 结果

**正面**：

- 病程可靠聚合，时间线"按健康问题视图"真正可用
- 接近 EMR Problem List，医疗建模成熟
- schema 长期稳定，不因标签字符串混乱而需清洗

**负面**：

- 增加一张表 + 一张关联表，查询需 join
- MVP 需要 AI 推荐关联逻辑（额外开发，但价值高）

## 备选方案

- **problem_tag TEXT（v2）**：已否决（同义词泛滥、无法聚合）
- **events.problem_id 单外键**：放弃多对多——但一个事件确实可能属于多个健康问题（发烧 + 哮喘随访同时），单外键表达不足
- **纯 tags 数组**：同 problem_tag TEXT 的问题
