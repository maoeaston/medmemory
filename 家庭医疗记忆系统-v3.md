## v3 主要变更（相对 v2）

1. **数据模型重构**（架构师视角，未来最易推翻重来的地方，现在一次做对）：

3. **新增 Storage Adapter 抽象**：统一 saveFile/getFile/deleteFile/listFiles 接口，下接 IndexedDB / OPFS，未来接 OneDrive / iCloud / WebDAV（详见 9.2）
4. **新增 Dashboard 医疗首页（7.1）进 MVP**：打开 App 第一眼看到成员慢病 + 当前用药 + 药箱过期预警，高频触点集大成（详见 7.1）

7. **路线图新增第三阶段**：疫苗/免疫接种档案、成长记录、用药历史时间轴（详见 12.4）

## 1. 产品定位

### 产品名称

家庭医疗记忆系统（Family Health Memory）

### 产品愿景

建立一个长期积累的家庭医疗记忆库。

让家庭成员过去的就诊记录、检查报告、化验单、处方信息、药品档案，能够在几年后依然快速检索和查看。

**产品哲学**：医疗领域里，**记忆比推理更有价值**。家庭真正需要的往往不是 AI 给出的诊断，而是 3 年前那份检查报告、那次药物过敏记录、那张出院小结。本系统的核心价值是长期可靠的医疗记忆，AI 只是整理记忆的工具。

---

## 2. 产品目标

解决家庭医疗信息分散的问题。

### 当前现状

医疗资料散落在微信聊天记录、手机相册、PDF 文件、医院 APP、纸质报告，导致找资料困难、重复检查、忘记病史、忘记用药记录。

### 产品目标

构建统一的家庭医疗档案库，实现：

- 医疗资料统一归档
- 医疗事件时间线管理
- 药箱清单管理（MVP 极简版）/ 完整药品档案（第二阶段）
- AI 辅助摘要 + OCR 全文
- AI 辅助搜索（第二阶段）

---

## 3. 使用场景与约束

### 3.1 使用场景

- **当前定位**：单家庭自用，由 1～2 名家庭管理员维护
- 不面向多用户、不对外发布、不商业化
- 数据规模：家庭成员 2～6 人，累计事件数百到数千条，原件数千份

### 3.2 现阶段取舍：AI 接入优先于隐私合规

**决策**：当前阶段优先保证 AI 能力可用，不投入精力做隐私合规框架。

**依据**：自用场景下数据敏感度评估可放宽；AI 摘要的实用价值高于隐私合规成本。

**含义**：

- 上传的医疗资料（图片/PDF）将直接发送至第三方 AI 服务完成 OCR、多模态识别、摘要
- 图片上的姓名、身份证号、就诊卡号等 PII 会随原图一起发送至 AI 服务（自用场景已接受上云）
- 通过 prompt 约束 AI 不把 PII 写入摘要和标签，降低结构化数据泄露面（PII 上云不可避免，但不进本地数据库）
- 不做数据流向弹窗、不做本地 OCR 模型兜底、不做端到端加密
- AI 服务选型以"接入简单、效果好"为标准

**重新评估时机**（必须触发）：

- 若未来分享给亲友或公开发布
- 若成员资料涉及非直系亲属明示不同意上云

### 3.3 数据安全底线（与隐私无关，必须做）

即便自用，以下仍是不可妥协的底线，原因是它们与隐私无关，而是"数据不应丢失"的工程基本面：

- 数据本地存储为主，避免单点丢失
- 支持完整导出，可迁移到其他设备
- 浏览器清缓存、换手机、手机丢失场景下，数据可恢复

详见 7.7 数据持久性。

## 4. 产品边界

### 本产品不做

- **不做在线问诊**：不替代医生
- **不做疾病诊断**：不输出疾病诊断结果
- **不做自动开药**：不推荐处方药
- **不做药品销售**：不接入电商平台
- **不做医院系统对接**：不依赖医院接口
- **不做全国医院格式适配**：保留原始文件即可
- **不做多用户权限/分享**（当前阶段自用，不引入账号体系）

---

## 5. 核心设计原则

### 原则一：原件优先

所有资料保留原始文件（JPG / PNG / PDF）。AI 解析失败不影响查看。

### 原则二：医疗事件驱动

系统核心对象是 **Medical Event（医疗事件）**，接近医院 EMR 的 Encounter 模型。例如：

```
2026-06-18 大宝发烧
├─ 门诊病历
├─ 血常规
├─ 处方
└─ 药品
```

### 原则三：AI 只负责整理

AI 职责：OCR 识别、内容摘要、标签提取、搜索辅助。
AI 不负责：诊断、用药建议、医疗决策。

### 原则四：AI 内容可信度标注

- 所有 AI 生成内容必须显著标注 **"AI 生成，以原件为准"**
- 关键数值（剂量、检验指标、参考范围、日期、药品名）在摘要中**保留原文**，不做改写、不做单位换算
- 摘要下方提供一键跳转原件的入口
- 依据：医疗数据 AI 出错代价高（误读剂量、误判指标），必须让用户能快速回到原件核对

### 原则五：数据架构面向长期（v3 新增）

数据模型和存储架构是未来最易推翻重来、代价最大的地方。现在多花时间把 schema 设计对（独立表、多对多、避免脏字段），比 MVP 跑起来后重构划算。

## 6. 用户角色

### 家庭管理员

- 自用场景下通常 1 人，最多 2 人（如夫妻）

### 7.1 医疗首页 Dashboard（v3 新增，MVP）

**定位**：App 打开第一眼看到的页面。高频触点的集大成——整合已有数据，不增加新数据采集负担，但极大提升日常打开价值。

**为什么进 MVP**：用户打开 App 不是每次都为了录病历（低频深度），更多时候是"快速看一眼家里医疗状态"（高频轻量）。Dashboard 把成员慢病、当前用药、药箱预警聚合到首屏，是比 AI 搜索更高频、更有价值的入口。

**首屏内容**：

### 7.3 医疗事件管理（v3 升级：健康问题多对多关联）

#### 字段

| 字段 | 说明 |
|------|------|
| 日期 | 就诊日期 |
| 成员 | 所属成员 |
| 医院 | 医院名称 |
| 科室 | 科室 |
| 标题 | 事件标题 |
| 类型 | 就诊类型 |
| **健康问题** | v3 升级：改为关联 `health_problems` 表（多对多），不再用 problem_tag TEXT |
| 备注 | 补充说明 |

#### 事件类型

门诊、急诊、体检、复查、疫苗、住院、其他。

#### 健康问题关联（v3 重构）

事件可关联 0 个或多个健康问题（来自 `health_problems` 表），通过 `event_problem_rel` 多对多关联。

**为什么不用 problem_tag TEXT**：几年后一定会出现"发烧/发热/高热""上呼吸道感染/呼吸道感染/反复呼吸道感染"等同义词泛滥，无法聚合病程。独立表 + 关联才能形成真正的病程追踪。

**示例**：

```
日期：2026-06-18
成员：大宝
标题：发烧39度
医院：儿童医院
科室：呼吸科
健康问题：[反复呼吸道感染] [哮喘随访]   ← 关联到 health_problems 表
```

### 7.4 医疗资料归档（v3 升级：storage_key）

#### 支持上传

JPG / PNG / PDF

#### 资料类型

门诊病历、检查报告、化验单、出院小结、收费单据、医嘱单。

#### 存储内容

| 字段 | 说明 |
|------|------|
| 文件名称 | 原始名称 |
| 上传时间 | 自动记录 |
| 所属事件 | Event ID |
| **storage_key** | v3 改名：存储抽象键（如 `attachment_83291`），实际位置由 Storage Adapter 管理 |
| 标签 | v3 新增：最新标签快照（JSON 数组） |
| AI 产出 | 关联 ai_contents 表（摘要、OCR 全文等） |

**为什么用 storage_key 不用 file_path**：PWA 下文件存在 IndexedDB 或 OPFS，根本不存在稳定的文件路径。直接存路径会把存储实现耦合进数据模型，未来从 IndexedDB 换 OPFS 或接网盘时要痛苦地清洗数据。用抽象 key + Storage Adapter，存储层可独立替换。

### 7.5 AI 摘要与 OCR 全文（v3 升级：OCR 全文 + 多版本）

上传资料后自动执行。v3 不只生成摘要，还保留原始 OCR 全文，且都支持多模型多版本。

#### AI 产出类型（存 ai_contents 表）

| content_type | 内容 | 用途 |
|------|------|------|
| `summary` | 结构化摘要 | 事件页快速浏览 |
| `ocr_fulltext` | 原始 OCR 全文 | 未来 RAG 搜索、全文检索的基础 |

**为什么存 OCR 全文**：未来做 RAG 或全文搜索时，如果只存摘要，几千份文件要重新跑 OCR——非常痛苦。现在顺手存下来，零额外 API 成本（同一轮多模态调用即可产出），未来值钱。

#### 摘要输出格式（示例：血常规报告）

```
【AI 摘要】⚠️ AI 生成，以原件为准

关键指标：
- 白细胞计数 11.2 ×10⁹/L（参考 4.0～10.0，偏高）
- 中性粒细胞比例 78%（参考 50%～70%，偏高）

文字部分摘要：
- 提示存在感染可能
- 建议结合临床和医生诊断

标签：发热、感染、血常规

[ 查看原件 ] [ 查看 OCR 全文 ] [ 重新生成 ]
```

#### 强制要求

- 顶部固定标注 **"AI 生成，以原件为准"**
- 数值类信息保留原文（数值、单位、参考范围不转换、不改写）
- **摘要和标签中不得包含姓名、身份证号、就诊卡号等 PII**（由 prompt 约束；原图 PII 已上云，但不进本地数据库）
- 摘要下方提供"查看原件"和"查看 OCR 全文"入口
- 提供"重新生成"入口，AI 误读时用户可重新触发（旧版本保留在 ai_contents 表，可对比）

### 7.7 数据持久性（v3 加分卷预留）

即便自用，数据丢失不可接受。浏览器 IndexedDB 在清缓存、换设备、手机丢失时会全没，必须有兜底。

#### 关键原则

备份必须包含原件（PDF/JPG）。原件也存在 IndexedDB/OPFS 里，只导出 SQLite 数据库等于只恢复元数据，原件丢了就永久丢了。因此备份永远是"数据库 + 原件 + 元数据"一起打包。

#### 功能

- **一键导出**：整个档案库打包为 ZIP
  - 包含：所有原件文件 + SQLite 数据库文件 + JSON 格式元数据
  - 文件名：`medmemory-backup-YYYYMMDD-HHmm-{N}成员-{M}事件.zip`
  - 示例：`medmemory-backup-20260621-1830-4成员-127事件.zip`
- **一键导入**：从 ZIP 恢复到当前设备
  - 导入前校验 ZIP 结构和版本兼容性
  - 提供合并/覆盖两种模式
  - **导入前自动快照**：执行导入（尤其覆盖模式）前，系统先自动把当前档案库备份成一份 ZIP，再执行导入。误操作可逆——选错文件或 ZIP 损坏时，可从快照恢复
- **导出后完整性校验**：导出完成后自动校验 ZIP 内原件文件数 = 数据库 attachments 记录数，不一致时立即告警
- **未备份提醒**：满足以下任一条件时，首页和设置页提示"建议备份"（非强制弹窗）：
  - 距上次导出超过 30 天
  - 累计新增/修改事件 ≥ 20 条且未导出
- **手动触发**：设置页提供"立即备份"按钮
- **建议频率**：每月或每次大量录入后导出一次

#### 未来扩展（不进 MVP）

- **分卷导出**：几年后数据可能达 5GB+，浏览器一次性打包大 ZIP 可能崩溃；预留按卷（如每 500MB）分批导出能力
- 自动定时备份到网盘路径（OneDrive / 坚果云 / WebDAV，依赖 Storage Adapter）
- 增量备份

### 7.8 药箱清单与药品档案（MVP 极简版 + 第二阶段增强）

本模块分两阶段。MVP 先上**极简药箱清单**作为高频触点，第二阶段再做完整药品档案。

**为什么药箱进 MVP**：病历检索是低频深度需求，"这个药还有没有/放哪了/过期没"是高频轻量需求。app 撑过 6 个月持续使用，往往靠高频小需求把人不断带回来。

---

#### MVP：极简药箱清单

**定位**：高频轻量触点，不做状态机、不做提醒推送。

**功能**：

- 纯手动维护的药品列表
- 字段：药品名称、有效期（YYYY-MM）、存放位置、备注、归属成员（可空表示家庭共用）
- **过期高亮**（被动展示，不推送）：
  - 有效期已过 → 列表项标红
  - 有效期 ≤ 30 天 → 列表项标黄
  - 其余正常显示
- 支持按归属成员筛选、按有效期排序（快过期的排前面）

**示例**

```
药品：美林
位置：客厅电视柜第二层
有效期：2027-08
归属：大宝
备注：儿童退烧备用
```

#### 第二阶段增强

- **拍照识别药盒**：拍摄药盒，多模态 AI 自动填药品名称
- **与事件关联**：source_event_id，处方药关联到就诊事件
- **过期提醒推送**：90 天 / 30 天 / 已过期主动提醒（区别于 MVP 的被动高亮）

## 9. 技术方案

### 9.1 前端

Vue3 + TypeScript + PWA

### 9.2 本地存储与 Storage Adapter（v3 重构）

#### SQLite 持久化：直接定 sqlite-wasm + OPFS（v3 砍掉 SQL.js）

v2 曾保留 SQL.js + OPFS 对比，v3 直接判 SQL.js 死刑。

**为什么不用 SQL.js**：SQL.js 不支持增量写入，每次 insert/update 后必须 `db.export()` 把整个数据库序列化成 blob 整体存回 IndexedDB。按 4 人家庭 10 年约 800 事件 × 5 附件 = 4000 附件，叠加 AI 摘要和 OCR 全文长文本，每次保存的整体序列化会越来越慢。这个坑已知且无解，没必要再花阶段 0 验证。

**定 sqlite-wasm + OPFS（私有源文件系统）VFS**：

- 支持增量持久化，无需整体序列化，长期性能稳定
- 兼容性：阶段 0 验证目标浏览器（Chrome/Edge 现代版支持良好；Safari 较新版本支持；需实测）

阶段 0 只验证 sqlite-wasm + OPFS 一条路径，不再对比 SQL.js。

#### Storage Adapter（v3 新增基础设施）

抽象存储层，数据模型只持有 `storage_key`，不关心文件实际存在哪里。

```typescript
interface StorageAdapter {
  saveFile(key: string, blob: Blob): Promise<void>
  getFile(key: string): Promise<Blob | null>
  deleteFile(key: string): Promise<void>
  listFiles(prefix?: string): Promise<string[]>
}
```

**当前实现**：IndexedDB（原件文件）、OPFS（SQLite 数据库文件）

**未来实现**（不进 MVP，但接口已预留）：OneDrive、iCloud、WebDAV——届时只需新增 Adapter 实现，数据模型零改动。

**为什么现在就抽象**：评审指出这是未来最易推翻重来的地方。如果数据模型直接耦合 IndexedDB，未来接网盘要痛苦地清洗数据。现在用一层薄抽象，成本极低，收益是存储层可独立演进。

### 9.3 AI 能力

#### 推荐方案：单一多模态大模型 API

一个模型同一轮调用产出摘要 + OCR 全文 + 标签，省去单独接 OCR 服务。

**候选模型**（按推荐度，最终选型在阶段 0 验证）：

- **GPT-4o / GPT-4o-mini**：多模态成熟、生态好、调用简单
- **Claude Sonnet 4.5**：文字摘要质量高、长文档处理强
- **Gemini 2.0 Flash**：性价比高、速度快

**调用方式**：图片 base64 + 固定 prompt 模板，prompt 约束包括：

- 保留原文数值（剂量、指标、单位、参考范围、日期、药品名不转换、不改写）
- 不输出诊断结论、不给出用药建议
- 输出结构化标签
- **不把姓名、身份证号、就诊卡号等个人身份信息写入摘要和标签**（原图 PII 仍会上云，但摘要库/数据库不留存结构化 PII，降低泄露面）
- 同时输出摘要和 OCR 全文（一次调用两份产出，存入 ai_contents 表）

**成本预估**：自用调用量小（每月数十到数百次），月成本可控在数美元量级。

#### 第二阶段再加

- RAG 搜索（基于已存的 ocr_fulltext 做 embedding + 向量库）
- 拍照识别药盒（复用多模态能力）
